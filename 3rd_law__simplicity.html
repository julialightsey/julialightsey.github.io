If a method is too complex to effectively unit test, it is too complex.

Simplicity is the ultimate sophistication. Leonardo da Vinci.

If you can't test it, it doesn't exist. It's really that simply. Any thing else is supposition, postulation, and theory. It is never my preference to try to promote a theoretical object into production, yet that is exactly what the majority of software programmers do daily.

In just six weeks from the time the design was started, we had the motor on the block testing its power. Orville Wright

Some believe they have fulfilled the obligations of testing by checking the "unit test" check box in Rally. Most believe that it is possible for them as individuals to determine by inspection that their work meets requirements. Others believe that it is possible for a peer to review their code and to determine by inspection that their work meets requirements.

The problem is that 100% inspection is only 70% efficient. The only effective way to fully test that software meets requirements is through the use of programmatic tests.

Complexity is generally used to characterize something with many parts where those parts interact with each other in multiple ways. The study of these complex linkages is the main goal of complex systems theory.

In science, there are at this time a number of approaches to characterizing complexity...

Neil Johnson admits that "even among scientists, there is no unique definition of complexity - and the scientific notion has traditionally been conveyed using particular examples..." Ultimately he adopts the definition of 'complexity science' as "the study of the phenomena which emerge from a collection of interacting objects."

One of the primary goals of any design is to reduce its complexity. Complexity is not a desired artifact of any system, but rather, complexity must justify itself.

Chaos: When the present determines the future, but the approximate present does not approximately determine the future. Edward Lorenz.

Many development shops have standards for complexity. Some do not and you will find stored procedures that are three to five thousand lines long and are just a?big ball of mud, sometimes referred to as spaghetti code. Some shops simply state that a method should be no longer than a certain number of lines. This is valid at face value, but you can rapidly find that you are having to create multiple methods to perform accessor and mutator operations on the same object. By using an arbitrary limitation as a complexity rule, you have forced the complexity into the public interface. A complexity measure that is common with the?Agile methodologies(and one which I carry over from my time working with dangerous machinery) is that a method is too complex if it cannot be effectively tested programmatically. This is a core and driving factor in the?Test Driven Development methodology.

The logic behind this statement is simple; if you can effectively, repeatably, programmatically, and fully test an object, it can be refactored?in the future for performance, extension, clarity, etc. If you cannot effectively, repeatably, programmatically, and fully test an object, then even fifteen lines of code is too long. If it can be tested, you can modify or enhance it and run the original tests against it to ensure that the original functionality has not been lost. If you cannot test it, you have to wait for the customer to tell you if you have impacted the original functionality. In 'machinery' terms, that would mean that I would have to wait until someone is killed or maimed to find out if my design change is valid. In business software terms, that means I may have to wait until a hacker steals or corrupts business data, or until a client?s data is compromised or corrupted to know that my design change is valid. Neither option is acceptable.

Note that nothing is said here about the test environment. That is not an indication that the test environment or the process of change management is of minor importance. To the contrary, the test environment and the change management process can be the deciding factor between success and failure. Nonetheless, both are supporting components of the design and build process rather than primary components.

The most common means of defining complexity of software methods are by lines of code and by number of operations performed.

  1. The number of lines of code in an object is an easy measure, and thus an easy target, but is an arbitrary measure at best. Formatting differences can dramatically change the number of lines of code, and attempts to condense code into fewer lines can actually add to the complexity.

    Consider the following two sets of code. Syntactically they are identical. The first requires only one line. The second requires six lines, yet it is significantly easier to read. Code reviews that use the length of an object measured by lines of code force programmers into the first method, increasing actual rather than perceived complexity and reducing the ability of the programmer to maintain or update the object in the future.

    Insisting on limiting the number of operations in an object may force the object to broken into constituent parts that do not make sense on their own.

    I prefer to encapsulate the complexity into an object that can be reused and refactored if necessary while simplifying both the architecture and the calling objects.

The counterpoint to this is that you must avoid creating a "God Object". This is easily avoided by writing proper unit tests (see: Test Driven Development). In the case of the procedure above, tests can clearly be written for each case; insert, update, delete, verbose output, and logging.

Simplicity is a great virtue but it requires hard work to achieve it and education to appreciate it. And to make matters worse: complexity sells better. Edsger Wybe Dijkstra.

This is a difficult concept for those who have not written unit tests. Write a few using the methodology presented in Test Driven Development and you will rapidly see that unit tests easily constrain complexity.

The two methods that should be used to constrain complexity on an object are:

  1. Reduce complexity in the architecture and in calling objects.
  2. All interactions of the object as well as side effects must be testable by programmatic unit tests.

The best means of managing complexity is the unit test. If tests can be written that effectively test the required business functionality of the object, the object can be improved and augmented now or at a later date. If tests cannot effectively determine that the object meets required business functionalilty, then it is simply too complex.

If the software under inspection is simply too complex to be unit tested, it should be refactored until the individual components may be unit tested. Unit testing serves in this manner to ensure both code compliance as well as code simplicity. Keep in mind that this is not an esoteric objective.

Code which is too complex to unit test will be too complex to maintain and extend as well.

/* 
  uses one procedure to get addresses by any of the variables passed in.  
  all the cases are not built out... it should be obvious that there would  
  be a cte for each of the parameters and all would be unioned together. 
   
  I do not mind using "select *" in this case as the select is from the cte 
  just preceding the select statement, and part of the overall construct. 
   
  This gets all of the addresses that meet any of the parameters. a technique 
  to get only addresses that meet all of the parameters is shown elsewhere. 
   
  There is only one method for the calling application to be aware of. This 
  is NOT an example of a God object (anti pattern) as the method still does 
  one very clearly defined task; get an address.

if object_id(N'[address].[get]', N'P') is not null
  drop procedure [address].[get];
go
create procedure [address].[get]
  @id                [int] = null
  , @street_01       [nvarchar](250) = null
  , @street_02       [nvarchar](250) = null
  , @rural_route     [nvarchar] (250) = null
  , @post_office_box [nvarchar](250) = null
  , @city            [sysname] = null
  , @state           [sysname] = null
  , @postal_code     [nvarchar](50) = null
as
  begin
      with [get_by_street_01]
           as (select [id]
                      , [street_01]
                      , [street_02]
                      , [rural_route]
                      , [post_office_box]
                      , [city]
                      , [state]
                      , [postal_code]
               from   [address].[data]
               where  [street_01] = @street_01),
           [get_by_street_02]
           as (select [id]
                      , [street_01]
                      , [street_02]
                      , [rural_route]
                      , [post_office_box]
                      , [city]
                      , [state]
                      , [postal_code]
               from   [address].[data]
               where  [street_02] = @street_02),
           [get_by_city]
           as (select [id]
                      , [street_01]
                      , [street_02]
                      , [rural_route]
                      , [post_office_box]
                      , [city]
                      , [state]
                      , [postal_code]
               from   [address].[data]
               where  [city] = @city),
           [union_all]
           as (select *
               from   [get_by_street_01]
               union
               select *
               from   [get_by_street_02]
               union
               select *
               from   [get_by_city])
      select [id]
             , [street_01]
             , [street_02]
             , [rural_route]
             , [post_office_box]
             , [city]
             , [state]
             , [postal_code]
      from   [union_all];
  end;
go
